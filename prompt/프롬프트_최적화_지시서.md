# 협력적 수업설계 카드 프롬프트 최적화 지시서

본 문서는 각 카드의 대화 응답 속도와 품질을 동시에 높이기 위한 **리소스 주입 최소화 원칙**과 **확인 멘트 개별화 규칙**을 정의한다. 핵심 아이디어는 간단하다.

> "카드가 정말 필요한 리소스만 프롬프트에 실어 보낸다."

불필요한 리소스(성취기준 DB 전체, 장문의 참고자료, 웹검색 결과 등)는 대화 속도와 토큰 비용에 직접적인 부담을 준다. 따라서 각 카드가 **실제로 수행하는 과제**에 맞춰 리소스를 선별하여 주입한다.

---

## 1. 리소스 인벤토리

시스템이 제공 가능한 리소스를 다음 네 가지로 분류한다.

| 코드  | 리소스                        | 무엇을 담고 있나                                              | 토큰 비용 |
| ---- | ----------------------------- | ------------------------------------------------------------ | -------- |
| `R1` | 2022 개정 교육과정 성취기준    | 교과별 성취기준 코드·원문·해설                                 | 높음     |
| `R2` | 교과별 핵심 아이디어 요약      | 단원별 핵심 아이디어 서술                                     | 중간     |
| `R3` | 사용자 업로드 참고자료         | 사용자가 프로젝트에 첨부한 문서(PDF·hwpx·docx 등)              | 가변     |
| `R4` | 웹검색 스니펫                 | 외부 최신 사례·통계·뉴스                                      | 가변     |
| `R5` | 본 프로젝트의 선행 카드 요약   | 같은 프로젝트에서 이미 작성된 카드들의 요약(역할·일정·문제 상황 등) | 낮음     |
| `R6` | 팀 구성원 프로필             | 각 교사의 과목·경력·관심 주제                                 | 낮음     |
| `R7` | 협력적 수업설계 방법론 가이드 | T·A·Ds·DI·E 단계 설명, 예시 문구                              | 낮음(캐시) |

**원칙**

- 토큰 비용 "높음" 리소스(R1, R3, R4)는 **필요한 카드에만** 붙인다.
- "낮음(캐시)"으로 표시된 R7은 시스템 프롬프트 레벨에서 상시 캐시한다.
- R5는 항상 붙이되 **요약본**(카드당 ≤ 200자)만 붙인다. 원문 전체 주입 금지.
- R3, R4는 사용자가 해당 카드에서 실제로 활용하도록 설계된 경우에만 주입한다.
- R1은 "성취기준이 의미 있게 변환되는 카드"에만 주입한다.

## 2. 카드별 리소스 주입 매트릭스

| 단계 | 카드        | 과제 요지                              | 주입 리소스                              | 사용 금지 리소스         |
| ---- | ----------- | ------------------------------------- | --------------------------------------- | ----------------------- |
| T    | T-1-1 공동 비전     | 팀의 방향을 한 문장으로 정리          | R5(선행요약), R6(팀 프로필), R7             | R1, R2, R3, R4          |
| T    | T-1-2 설계 방향     | 3~5개 방향 불릿 도출                  | R5, R6, R7                              | R1, R2, R3, R4          |
| T    | T-2-1 역할 배분     | 이름×역할 표 구성                     | R5, R6                                  | R1, R2, R3, R4, R7      |
| T    | T-2-2 팀 규칙       | 합의 문장 불릿 생성                   | R5, R6                                  | R1, R2, R3, R4          |
| T    | T-2-3 단계별 일정   | 일정 표                               | R5                                      | R1, R2, R3, R4, R6      |
| A    | A-1-1 주제 선정 기준 | 기준 불릿 도출                        | R5, R7                                  | R1, R3, R4              |
| A    | A-1-2 주제 선정     | **핵심 카드** · 최종 주제 확정        | **R1, R2, R3, R4**, R5, R7              | (없음)                  |
| A    | A-2-1 핵심 아이디어·성취기준 | 교과별 핵심 아이디어·성취기준 매핑 | **R1, R2**, R5                          | R3, R4                  |
| A    | A-2-2 통합 수업 목표 | 한 문장 학습자 진술문 도출             | R5(A-2-1 요약 필수), R2                   | R1 전체, R3, R4         |

| Ds   | Ds-1-2 문제 상황    | 학교 내 실제 상황 나열               | R3(사용자 업로드한 학교 현황), R5           | R1, R2, R4              |
| Ds   | Ds-1-3 학습자 활동  | 차시별 활동 목록                      | R5(문제 상황·성취기준 요약), R7              | R1 전체, R3, R4         |
| Ds   | Ds-2-1 지원 도구    | 활동 단계별 도구                      | R5, R7                                  | R1, R2, R3, R4          |
| Ds   | Ds-2-2 스캐폴딩     | 활동별 스캐폴딩                       | R5, R7                                  | R1, R2, R3, R4          |
| DI   | DI-1-1 개발 자료    | 자료 목록·검토자                      | R5, R6                                  | R1, R2, R3, R4          |
| DI   | DI-1-2 검토 유의점  | 체크 포인트 불릿                      | R5, R7                                  | R1, R2, R3, R4          |
| DI   | DI-2-1 실행 일정    | 차시×날짜×장소                        | R5, R6                                  | R1, R2, R3, R4          |
| DI   | DI-2-2 결과 확인 자료 | 차시별 산출물                        | R5                                      | R1, R2, R3, R4          |
| E    | E 체크리스트         | 종합 평가 점검                        | R5(모든 단계 요약), R7                    | R1, R2, R3, R4          |

**해석 예시**

- **A-1-2 주제 선정**은 네 가지 고비용 리소스(R1·R2·R3·R4) 모두를 사용한다. 이 카드는 수업 전체의 방향을 결정하는 분기점이므로 품질이 우선이다.
- **T-2-1 역할 배분**이나 **T-2-3 단계별 일정**은 성취기준·핵심 아이디어·참고자료·웹검색이 모두 불필요하다. 오직 팀 프로필과 선행 카드 요약만 사용한다. 응답 시간이 가장 짧아야 한다.
- **Ds-1-2 문제 상황**은 사용자가 업로드한 학교 현황 자료(R3)가 있을 때 그 일부만 발췌하여 주입한다. 웹검색(R4)은 명시적으로 사용하지 않는다.

## 3. 리소스 주입 규칙 상세

### 3.1 R1 (성취기준) 주입 규칙

- A-1-2, A-2-1 두 카드에만 전체 주입한다.
- 다른 카드에서 성취기준 코드를 언급해야 할 때는 **R5(선행 카드 요약)** 에서 코드만 인용한다.
- 주입 시에도 학년·교과 필터를 선행 적용한다. (예: 중2 · 과학/사회/국어만)

### 3.2 R2 (핵심 아이디어) 주입 규칙

- A-1-2, A-2-1, A-2-2 세 카드에만 주입.
- 주입 범위는 해당 프로젝트의 교과 목록으로 한정한다.

### 3.3 R3 (사용자 참고자료) 주입 규칙

- A-1-2와 Ds-1-2 두 카드에 우선 주입한다.
- 다른 카드에서 참고자료가 필요하다는 판단이 서면, 해당 자료의 **요약 200자**만 주입한다(원문 금지).
- 파일이 복수라면 카드 과제와 관련도가 가장 높은 1~2개만 선택한다.

### 3.4 R4 (웹검색) 주입 규칙

- A-1-2 카드에서만 사용한다.
- 검색 쿼리는 `{프로젝트 주제 후보} 학교 사례 2024..2026` 형태로 고정 템플릿을 사용한다.
- 결과는 상위 3개 스니펫만 주입한다.

### 3.5 R5 (선행 카드 요약) 주입 규칙

- 모든 카드에 주입. 단 **요약만**. 최대 200자.
- 요약은 카드가 저장될 때 비동기로 사전 생성해 두고 캐시에서 읽는다.

### 3.6 R6 (팀 프로필) 주입 규칙

- T·DI 단계 카드에 주입. A·Ds·E에는 주입하지 않는다.

### 3.7 R7 (방법론 가이드) 주입 규칙

- 시스템 프롬프트 수준에서 캐시하여 모든 카드가 참조 가능하게 한다.
- 카드 프롬프트에 추가 주입하지 않는다(중복 방지).

## 4. 프롬프트 템플릿 골격

각 카드의 프롬프트는 동일한 골격을 따른다. 가변 블록은 위 매트릭스에 따라 포함/제외된다.

```
[SYSTEM]
- 당신은 협력적 수업설계 도우미입니다.
- R7(방법론 가이드) 요약: {cached}
- 프로젝트 요약: {project_summary}

[RESOURCES]  ← 가변
{R1?} {R2?} {R3?} {R4?} {R5} {R6?}

[TASK]
카드: {card_code} {card_title}
과제: {one_line_task_description}
출력 스키마: {field_schema}
주의: {card_specific_rules}

[USER INPUT]
{user_free_text_or_previous_values}
```

**비용 최적화 체크리스트**

- [ ] 이 카드가 매트릭스에서 허용한 리소스만 들어갔는가?
- [ ] R3/R4는 200자 이하 요약으로 축약되었는가?
- [ ] R5(선행 요약)는 캐시에서 읽었는가?
- [ ] R1을 주입한 경우 학년·교과 필터가 적용되었는가?

## 5. 응답 마지막 확인 멘트 개별화

현재 모든 카드가 "이대로 ~ 진행하시겠습니까? 수정이 필요한 부분이 있으면 알려주세요." 문장으로 끝난다. 이는 대화 경험을 반복적이고 기계적으로 만든다. 각 카드의 과제 특성에 맞는 **차별화된 확인 멘트**를 사용한다.

규칙:

1. 모든 멘트는 두 문장 이내, 18~28자 정도의 자연스러운 한국어.
2. 첫 문장은 **확인/제안/합의** 중 하나의 어조, 두 번째 문장은 **사용자 수정 여지**를 열어 두는 안내.
3. 어미·동사를 카드 과제에 맞추어 다르게 고른다(예: 비전은 "다듬다", 역할은 "배분하다", 일정은 "확정하다", 평가는 "정하다", 문제 상황은 "좁히다" 등).
4. 하나의 카드에는 2~3개의 변형을 두어 같은 카드를 다시 수정할 때도 반복감이 적도록 한다.

### 5.1 카드별 확인 멘트 예시 (각 카드 최소 2개)

**T-1-1 공동 비전**
- "이 비전으로 팀의 방향을 잡아도 괜찮을까요? 더 담고 싶은 뉘앙스가 있으면 말씀해 주세요."
- "비전 문장이 여기까지 다듬어졌습니다. 고치고 싶은 표현이 있다면 편하게 알려 주세요."

**T-1-2 수업설계 방향**
- "이 방향들로 큰 줄기를 잡겠습니다. 추가하거나 덜어 낼 항목이 있으면 알려 주세요."
- "이 정도면 설계의 원칙이 분명해 보입니다. 더 강조하고 싶은 원칙이 있다면 말씀해 주세요."

**T-2-1 역할 배분**
- "이 구성으로 역할을 나눠 보겠습니다. 조정이 필요하면 어떤 선생님 역할을 바꿀지 알려 주세요."
- "이대로 역할 분배를 확정해도 될까요? 한두 자리 조정이 필요하면 말씀 주세요."

**T-2-2 팀 규칙**
- "이 합의사항을 팀 규칙으로 고정하겠습니다. 덧붙이고 싶은 약속이 있으면 알려 주세요."
- "이 정도 규칙이면 운영에 무리가 없어 보입니다. 수정이나 추가할 항목이 있을까요?"

**T-2-3 단계별 일정**
- "이 일정으로 전체 흐름을 확정해도 될까요? 날짜나 기한을 바꾸고 싶은 곳이 있다면 알려 주세요."
- "단계별 마감일을 이렇게 잡아 두었습니다. 당기거나 미뤄야 할 일정이 있으면 편하게 말씀해 주세요."

**A-1-1 주제 선정 기준**
- "이 기준들로 후보를 걸러 보겠습니다. 더하고 싶은 관점이 있다면 말씀 주세요."
- "선정 기준이 이 정도면 충분해 보입니다. 빠진 기준이 있으면 알려 주세요."

**A-1-2 주제 선정**
- "이 주제로 수업 전체 설계를 이어가도 괜찮을까요? 다른 방향이 떠오르시면 알려 주세요."
- "최종 주제 후보를 이렇게 좁혔습니다. 조금 더 다듬거나 다른 후보를 고려하고 싶다면 말씀 주세요."

**A-2-1 교과별 핵심 아이디어·성취기준**
- "핵심 아이디어와 성취기준을 이렇게 연결해 보았습니다. 누락되었거나 추가할 코드가 있다면 알려 주세요."
- "교과별 핵심 아이디어 매핑이 이 정도면 맞닿아 있어 보입니다. 보완할 지점이 있을까요?"

**A-2-2 통합 수업 목표**
- "이 문장을 통합 수업 목표로 두어도 될까요? 강조하고 싶은 부분이 있으면 다듬어 드릴게요."
- "학습자 수행을 한 문장으로 수렴해 보았습니다. 더 분명히 하고 싶은 표현이 있으면 알려 주세요."

**Ds-1-1 평가**
- "평가 주제·방법·루브릭을 이 구성으로 확정해도 될까요? 균형이 맞지 않는 축이 있다면 말씀해 주세요."
- "백워드 원칙에 따라 평가를 먼저 이렇게 잡았습니다. 조정이 필요한 항목이 있으면 알려 주세요."

**Ds-1-2 문제 상황**
- "학교 안에서 벌어지는 문제 상황을 이렇게 좁혔습니다. 다른 상황을 더 고려하고 싶다면 알려 주세요."
- "이 상황을 중심으로 수업을 전개해도 될까요? 보조로 쓸 상황도 함께 확인해 주세요."

**Ds-1-3 학습자 활동**
- "차시별 활동 흐름을 이렇게 짜 보았습니다. 순서나 활동 자체를 조정하고 싶다면 말씀해 주세요."
- "이 활동들로 수업 8차시를 구성해도 좋을까요? 손볼 활동이 있다면 알려 주세요."

**Ds-2-1 지원 도구**
- "각 활동에 붙일 도구를 이렇게 매칭해 두었습니다. 학교 환경에서 쓰기 어려운 도구가 있다면 알려 주세요."
- "이 도구 조합으로 진행해도 될까요? 대체하고 싶은 도구가 있으면 말씀 주세요."

**Ds-2-2 스캐폴딩**
- "활동별 스캐폴딩을 이렇게 배치했습니다. 학습자에게 더 필요한 도움이 있다면 알려 주세요."
- "이 정도 스캐폴딩이면 학습자 다양성을 어느 정도 품을 수 있을 것 같습니다. 보완 지점이 있을까요?"

**DI-1-1 개발 자료**

- 사용자 언어가 존댓말/반말에 따라 어미를 자동 조정한다(기본은 존댓말).

## 6. 구현 가이드 요약

1. **리소스 로더**를 카드 단위 함수로 분리한다: `load_resources(card_code) -> dict`. 매트릭스(섹션 2)가 이 함수의 진실의 원천.
2. **요약 캐시**를 Redis 또는 SQLite에 두어 R5를 200자 이하로 상시 유지한다. 카드 저장 시 백그라운드 작업으로 갱신한다.
3. **프롬프트 빌더**는 섹션 4의 골격을 따르되, 매트릭스에 없는 리소스 키는 키 자체를 누락시켜 토큰 비용을 영으로 만든다.
4. **확인 멘트 서비스**는 섹션 5의 카드×변형 맵을 담은 JSON을 읽어 임의 선택한다. 최근 사용 이력을 카드별로 기억해 직전 변형은 제외한다.
5. **비용 로깅**: 각 카드 호출마다 주입 리소스 크기를 누적 기록하여, 매트릭스를 분기마다 재조정한다.

## 7. 수락 기준

- 매트릭스에 명시되지 않은 리소스가 프롬프트에 주입된 카드 호출은 0건이다.
- A-1-2(주제 선정)와 T-2-1(역할 배분)의 평균 응답 시간 차이가 통계적으로 유의하게 후자가 빠르다.
- 사용자 확인 멘트가 동일 세션 내에서 두 번 연속 똑같이 반복되는 경우가 없다.
- 모든 카드의 확인 멘트가 "이대로 ~ 진행하시겠습니까? 수정이 필요한 부분이 있으면 알려주세요." 라는 고정 표현에서 벗어난다.
