# 협력적 수업설계 카드 재설계 지시서

## 배경

현재 협력적 수업설계 도구의 각 단계 카드는 대부분 단순 텍스트 입력창(textarea)으로 구성되어 있다. 최근 완성한 「협력적 수업설계안 보고서」(PDF)의 구조에 맞추어 카드의 입력 구조를 재설계하여, 카드에 쌓인 데이터가 **보고서 자동 생성의 직접 입력값**으로 사용될 수 있도록 한다.

## 전제 조건

- 각 카드 내부에 이미 존재하는 **"구성원 의견 수집 UI"는 그대로 유지**한다. 재설계 범위에 포함하지 않는다.
- 단계 코드(T·A·Ds·DI·E 및 하위 코드)와 카드 순서는 그대로 유지한다. **카드의 내용 필드(본문 입력 영역)만** 재설계한다.
- 각 카드의 저장 데이터는 JSON으로 직렬화되어 보고서 템플릿에 주입된다. 따라서 아래 스키마의 필드명을 준수한다.

## 공통 컴포넌트 타입

필요한 입력 컴포넌트 타입을 다음과 같이 정의한다.

- `text` — 한 줄 텍스트
- `textarea` — 여러 줄 텍스트
- `bullets` — 불릿 목록(행 추가/삭제 가능)
- `table` — 열 스키마가 고정된 표(행 추가/삭제 가능)
- `select` — 고정된 선택지 중 하나
- `tag` — 태그 입력(다중 선택)
- `date` — 날짜
- `richtext` — 볼드·강조를 포함한 짧은 서식 텍스트

표(table) 정의 형식:

```
table(columns=[
  { key, label, type: text|select|date|time, width?, options? },
  ...
], minRows, addable: true)
```

---

## 1단계. 팀 준비 (T)

### 카드 T-1-1 · 팀 공동 비전

- `vision` : `richtext` — 한 문장 형태의 공동 비전. 예: "학생들이 자기 삶의 공간에서 기후위기를 마주하고 …"
- `vision_note` : `textarea` — 비전의 배경·맥락 설명 (선택)

### 카드 T-1-2 · 수업설계 방향

- `directions` : `bullets` — 설계 방향 3~5개. 각 항목은 한 문장.

### 카드 T-2-1 · 역할 배분

필드를 단일 표로 변경:

- `roles` : `table(columns=[
  { key: name, label: "이름", type: text },
  { key: subject, label: "과목", type: text },
  { key: core_role, label: "핵심 역할", type: text },
  { key: area, label: "담당 영역", type: text }
])`

### 카드 T-2-2 · 팀 규칙

- `rules` : `bullets` — 합의 내용을 불릿으로 작성. (※ 기존 표 구조 제거)

### 카드 T-2-3 · 단계별 일정

- `schedule` : `table(columns=[
  { key: due_date, label: "목표 완료일", type: date },
  { key: content, label: "내용", type: text },
  { key: deliverable, label: "산출물", type: text }
], minRows: 5)` — **행 식별자는 '목표 완료일'**. '단계' 열은 두지 않는다.

---

## 2단계. 분석 (A)

### 카드 A-1-1 · 주제 선정 기준

- `criteria` : `bullets` — 주제 선정 기준 3~5개.

### 카드 A-1-2 · 주제 선정

- `candidates` : `bullets` — 후보 주제 목록
- `final_topic` : `text` — 최종 선정 주제 (한 줄, 강조 박스로 렌더링)
- `selection_rationale` : `textarea` — 최종 선정 사유

### 카드 A-2-1 · 교과별 핵심 아이디어 · 성취기준

교과 수만큼 행을 추가할 수 있는 표로 변경한다.

- `core_ideas` : `table(columns=[
  { key: subject, label: "교과", type: text },
  { key: core_idea, label: "핵심 아이디어", type: textarea }
])`
- `achievement_standards` : `table(columns=[
  { key: subject, label: "교과", type: text },
  { key: code, label: "성취기준 코드", type: text },
  { key: statement, label: "성취기준 내용(원문)", type: textarea }
])`

**중요**
- 성취기준 내용은 2022 개정 교육과정 원문 그대로 입력한다(요약 금지).
- 코드 예: `[9과17-01]`, `[9사(지리)08-02]`
- 성취기준 코드는 입력 시 `curriculum_standards.json`(교사용 검색 자원)에서 자동 추천 가능하도록 한다.

### 카드 A-2-2 · 통합 수업 목표 (2.4 연계 설명과 결합)

- `integration_narrative` : `textarea` — 핵심 아이디어 ↔ 성취기준 ↔ 협력적 수업설계의 연계 설명
- `integrated_goal` : `richtext` — 하나의 학습자 수행 진술문 형태의 통합 수업 목표 (강조 박스로 렌더링)

---

## 3단계. 설계 (Ds)

### 카드 Ds-1-1 · 평가 내용 및 방법

하나의 카드에 3개의 하위 섹션을 둔다.

#### 3.1.1 평가 주제
- `eval_questions` : `bullets` — "~할 수 있는가?" 형태의 평가 질문 3~4개.

#### 3.1.2 평가 방법
- `eval_methods` : `table(columns=[
  { key: type, label: "평가 유형", type: select, options: ["진단","형성","수행","총괄"] },
  { key: target, label: "평가 대상", type: text },
  { key: method, label: "평가 방법", type: text },
  { key: timing, label: "시점", type: text }
])`

#### 3.1.3 평가 기준 (수행 루브릭)
- `rubric` : `table(columns=[
  { key: axis, label: "평가 축", type: text },
  { key: level_4, label: "상 (4)", type: textarea },
  { key: level_3, label: "중 (3)", type: textarea },
  { key: level_2, label: "하 (2)", type: textarea }
])`

### 카드 Ds-1-2 · 문제 상황 아이디어 나열

**중요** 여기서의 문제 상황은 논의 과정에서 나온 이슈가 아니라, **주제(예: 기후위기)와 관련하여 학교 안에서 실제로 벌어지고 있는 문제**를 나열한다.

- `problem_situations` : `table(columns=[
  { key: situation, label: "학교 안의 문제 상황", type: textarea },
  { key: decision, label: "채택 여부", type: select, options: ["채택","보조 자료로 활용","보류","미채택"] }
])` — **번호 열은 두지 않는다**.
- `selection_rationale` : `textarea` — 채택 결정 근거

### 카드 Ds-1-3 · 학습자 활동 아이디어 나열

- `activities` : `table(columns=[
  { key: period, label: "차시", type: text },
  { key: activity, label: "학습 활동 아이디어", type: textarea },
  { key: linked_standards, label: "연결 성취기준", type: tag }
])`

### 카드 Ds-2-1 · 지원 도구

- `support_tools` : `table(columns=[
  { key: stage, label: "활동 단계", type: text },
  { key: tool, label: "도구 / 자원", type: text },
  { key: purpose, label: "활용 목적", type: text },
  { key: related_period, label: "관련 차시", type: text }
])` — **'관련 차시' 열을 필수로 포함한다**.

### 카드 Ds-2-2 · 스캐폴딩 방안

- `scaffolding` : `table(columns=[
  { key: activity, label: "활동", type: text },
  { key: plan, label: "스캐폴딩 방안", type: textarea }
])`

## 4단계. 개발·실행 (DI)

### 카드 DI-1-1 · 개발 자료 목록

- `dev_materials` : `table(columns=[
  { key: member, label: "팀원 이름", type: text },
  { key: material, label: "개발 자료", type: text },
  { key: content, label: "내용", type: textarea },
  { key: reviewer, label: "검토자", type: text }
])`

### 카드 DI-1-2 · 검토 유의점 및 고려사항

- `review_considerations` : `bullets` — 검토 시 확인할 항목들. 각 항목은 "키워드: 구체 질문" 형태 권장.

### 카드 DI-2-1 · 수업 실행 일정

- `exec_schedule` : `table(columns=[
  { key: period, label: "차시", type: text, align: center },
  { key: date, label: "날짜", type: date, align: center },
  { key: time, label: "시간", type: text, align: center },
  { key: place, label: "장소", type: text, align: center },
  { key: teacher, label: "담당 교사", type: text, align: center }
])` — **차시·날짜·시간·장소·담당 교사 열은 모두 가운데 정렬**.

### 카드 DI-2-2 · 수업별 수행 결과 확인 자료

- `outputs` : `table(columns=[
  { key: period, label: "차시", type: text, align: center },
  { key: output_desc, label: "수행 결과 확인 자료", type: textarea }
])`

---

## 5단계. 평가·성찰 (E)

### 카드 E-1-1 / E-2-1 · 협력적 수업설계 종합 평가 체크리스트

- `design_rubric` : `table(columns=[
  { key: area, label: "영역", type: select, options: ["T 팀 준비","A 분석","Ds 설계","DI 개발·실행","E 평가·성찰","공통"] },
  { key: question, label: "평가 문항", type: textarea },
  { key: score, label: "점수(1~4)", type: select, options: ["1","2","3","4"] }
])`
- `reflection_note` : `textarea` — 팀 성찰 메모 (개선 과제 기록용)

## 보고서 자동 변환 규칙

카드에 저장된 데이터를 보고서 PDF로 변환할 때 적용되는 규칙:

- 성취기준 코드는 **볼드** 처리하고, 코드와 원문 사이에는 반각 공백 한 칸.
- `bullets` 는 보고서에서도 불릿 리스트로 렌더링한다.
- `table` 은 보고서의 wrap_table 컴포넌트로 렌더링되며, 모든 셀은 자동 줄바꿈된다.
- `final_topic`, `integrated_goal` 과 같은 `richtext` + 강조 필드는 보라색 박스(SOFT_PURPLE + PURPLE 테두리)로 강조 렌더링.
- 표 열의 `align: center` 는 보고서 셀 정렬에도 그대로 적용된다.
- 3단계의 문제 상황 표는 번호 열을 렌더링하지 않는다.
- 1단계의 단계별 일정 표는 '목표 완료일'이 맨 앞에 위치한다.

## 구현 지시 요약

1. 카드 컴포넌트 타입 시스템(위의 공통 컴포넌트 타입)을 먼저 정의한다.
2. 각 카드의 **본문 입력 영역**만 위 스키마에 따라 교체한다. **구성원 의견 수집 UI는 건드리지 않는다**.
3. 저장 모델을 위 필드명으로 마이그레이션하되, 기존 프리텍스트 입력은 해당 카드의 첫 번째 `textarea`/`bullets` 필드로 자동 이전한다.
4. 보고서 PDF 생성 파이프라인은 본 스키마를 직접 입력으로 받아야 한다. 임의의 프리텍스트 파싱에 의존하지 않는다.
