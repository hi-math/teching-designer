# 협력적 수업설계 AI 에이전트 — MVP 설계안

> 이 문서는 중등 교사팀의 협력적 수업설계를 지원하는 AI 에이전트 웹 서비스의 MVP 기능 및 플로우를 정의합니다.

---

## 1. 서비스 개요

### 목적

초중등(K-12) 교사들이 팀을 이루어 프로젝트 기반 수업(PBL)을 협력적으로 설계할 수 있도록 단계별로 퍼실리테이션하는 AI 코칭 에이전트를 제공한다.

### 핵심 원칙

| 원칙 | 설명 |
|------|------|
| 상호 의존 | 팀 비전을 공유하고 신뢰 기반의 설계팀을 형성한다 |
| 인지 분산 | 팀원들의 역할과 인공물을 통해 인지를 사회적으로 분산한다 |
| 활성화 | 구성원이 자유롭게 아이디어를 생성하고 발언한다 |
| 외현화 | 개인의 생각을 시각적으로 표상하고 팀과 공유한다 |
| 조정 | 다양한 의견을 통합·보완하여 공동 산출물로 만든다 |

### 에이전트의 역할

AI는 설계의 주도자가 아니라 **퍼실리테이터**로서 다음 방식으로 개입한다.

- **안내**: 단계별 활동 방법 및 절차 안내
- **생성**: 초안, 예시, 키워드 군집화, 비교표 생성
- **점검**: 정합성 검토 (비전 ↔ 설계방향 ↔ 목표 ↔ 평가)
- **기록**: 활동 결과를 자동 저장하고 최종 리포트에 반영

## 2. 사용자 유형 및 역할

| 역할 | 설명 | 권한 |
|------|------|------|
| **Host** | 서비스 운영자, 에이전트 환경 관리자 | AI 액션 컨트롤 + 워크플로 순서 변경 + 설정 메뉴 접근 + 전체 편집 + 팀원 초대 + 출력 |
| **Editor** | 팀원 교사, 협력 설계자 | 설계 편집 + AI 채팅 |
| **Viewer** | 참관 또는 검토자 | 열람 + 코멘트 |

### Host 권한 상세

Host는 에이전트의 동작 방식을 전역적으로 설정하며, 개별 워크스페이스와 독립적으로 관리된다.

**AI 액션 컨트롤**
- 단계별로 AI 개입 유형(안내 / 생성 / 점검 / 기록) 활성화·비활성화
- 특정 단계에서 AI 자동 기록 기능 온/오프

**워크플로 순서 변경**
- 기본 5단계(T → A → Ds → DI → E) 순서를 재배열
- 특정 단계를 숨김 처리하거나 필수/선택으로 지정

### Host 설정 메뉴

| 설정 항목 | 설명 |
|-----------|------|
| API 키 입력 | LLM 호출에 사용할 API 키 등록 및 변경 |
| 모델 선택 | 사용할 LLM 모델 지정 (예: claude-sonnet, claude-opus 등) |

## 3. 핵심 기능 (MVP 범위)

### 3-1. 인증 및 로그 (AUTH & LOG DB)

#### 인증

- 이메일/패스워드 로그인
- 초대 링크 기반 팀 합류 (이메일 토큰 인증)
- 세션 관리 및 자동 로그아웃

#### 활동 로그 DB

모든 설계 활동의 변경 이력을 기록한다. 로그 항목은 다음 정보를 포함한다.

- 사용자 ID, 워크스페이스/프로젝트 ID
- 활동 유형 (생성 / 수정 / 삭제 / AI 수락 / AI 거절)
- 변경 전후 내용 (JSONB 스냅샷)
- 타임스탬프

활동 로그는 버전 히스토리 조회 및 최종 리포트 자동 생성의 기반이 된다.

---

### 3-2. LLM 챗봇

#### 채팅 인터페이스

- 설계 캔버스 우측에 상시 노출되는 사이드 패널 형태
- 스트리밍 방식으로 응답을 실시간 출력
- 대화 히스토리는 프로젝트 단위로 DB에 저장되어 세션 재접속 시 이어짐
- 메시지마다 발신자(교사 개인 / 팀) 구분 표시

#### AI 개입 유형

| 개입 유형 | 설명 | 예시 |
|-----------|------|------|
| 안내 | 해당 단계의 활동 방법 설명 | "비전 설정 단계에서는 ..." |
| 생성 | 초안·예시·키워드 군집 생성 | 팀 비전 후보 문장 3개 제안 |
| 점검 | 정합성 자동 검토 | 비전-목표 간 연결 여부 확인 |
| 기록 | 확정 내용을 설계 칸에 반영 | 수락 시 해당 섹션 자동 업데이트 |

#### AI 컨텍스트 구성

매 요청마다 다음 정보를 AI에게 전달한다.

- 현재 단계 코드 (예: T-1-1, A-2-2, Ds-1-3 등)
- 학습자 맥락 (학년, 과목, 디지털 리터러시 수준, 이전 수업 맥락)
- 팀 비전 및 설계 방향 (확정된 경우)
- 현재 설계 캔버스 전체 내용 (JSONB)
- 최근 대화 히스토리

#### AI 제안 수락 흐름

1. AI가 제안 내용을 채팅 버블로 출력
2. 교사가 **수락 / 수정 후 수락 / 거절** 중 선택
3. 수락 시 해당 설계 섹션에 자동 반영 + 버전 스냅샷 저장
4. 거절 시 대화 유지, 재요청 가능

### 3-3. 팀 활동 인터페이스

#### 워크스페이스 구조

```
워크스페이스 (팀 단위)
└── 프로젝트 (수업 단위)
    ├── 설계 캔버스 (단계별 섹션)
    ├── AI 채팅 패널 (공유)
    ├── 코멘트 스레드 (섹션별)
    └── 버전 히스토리
```

#### 팀 구성 흐름

1. Owner가 워크스페이스 생성 (학교명, 팀명 입력)
2. 이메일로 팀원 초대 + 역할 지정 (Editor / Viewer)
3. 팀원이 초대 링크 수락 → 워크스페이스 합류
4. 프로젝트 생성 (과목, 학년, 단원명 입력)

#### 설계 캔버스 — 단계별 섹션

워크플로우에 따라 다음 5개 단계로 구성된다. 각 단계는 독립적인 섹션으로 표시되며, 완료 여부를 체크할 수 있다.

| 단계 코드 | 단계명 | 핵심 산출물 |
|-----------|--------|-------------|
| T | 팀 준비 | 팀 비전, 설계 방향, 역할 분담, 팀 규칙, 팀 일정 |
| A | 분석 | 주제 선정 기준, 최종 주제, 성취기준 분석, 통합된 수업 목표 |
| Ds | 설계 | 평가 계획, 문제 상황, 학습 활동, 지원 도구, 스캐폴딩 방안 |
| DI | 개발/실행 | 개발 자료 목록, 수업 기록 |
| E | 평가/성찰 | 수업 성찰, 설계 과정 성찰 |

각 섹션은 다음 구성으로 표시된다.

- **활동 안내 텍스트**: 해당 단계의 목표 및 방법 (접을 수 있음)
- **입력 영역**: 개인 구상 → 팀 협의 결과 입력
- **AI 채팅 연동 버튼**: 해당 단계에 맞는 AI 프롬프트 자동 시작
- **저장 상태 표시**: 임시저장 / 확정 / 버전 잠금

#### 팀 협업 기능

**비동기 코멘트**
- 각 섹션에 코멘트 스레드 부착 가능
- 코멘트 등록 시 팀원에게 인앱 알림
- 코멘트 해결(resolve) 처리로 정리

**역할별 UI 분기**
- Owner/Editor: 전체 편집 가능
- Viewer: 편집 불가, 코멘트만 가능
- 각 섹션에 "마지막 수정자 + 시각" 표시

### 3-4. 활동 내용 출력

#### 출력 형식

| 형식 | 용도 |
|------|------|
| **Markdown (.md)** | 개발자·연구자 활용, 후속 편집 용이 |
| **PDF** | 공식 보고 및 인쇄 배포 |

#### 리포트 구성

출력물은 다음 섹션으로 자동 구성된다.

```
협력적 수업설계 결과 리포트
├── 표지 (워크스페이스명, 팀원, 생성일)
├── 1. 팀 준비
│   ├── 팀 비전
│   ├── 설계 방향
│   ├── 역할 분담
│   └── 팀 규칙 / 일정
├── 2. 분석
│   ├── 주제 및 선정 기준
│   ├── 성취기준 분석
│   └── 통합된 수업 목표
├── 3. 설계
│   ├── 평가 계획
│   ├── 문제 상황
│   ├── 학습 활동
│   ├── 지원 도구
│   └── 스캐폴딩 방안
├── 4. 개발/실행
│   ├── 개발 자료 목록
│   └── 수업 기록
└── 5. 평가/성찰
    ├── 수업 성찰
    └── 설계 과정 성찰
```

#### 출력 트리거

- **단계 완료 시**: 해당 섹션 내용이 리포트에 자동 반영
- **수동 출력**: 언제든 현재 상태 기준으로 출력 가능
- **최종 출력**: 전체 완료 후 종합 보고서 생성

## 4. 데이터 구조 (개념 모델)

```
users
  id, email, name, created_at

workspaces
  id, name, owner_id, created_at

members
  workspace_id, user_id, role (host / editor / viewer)

invites
  workspace_id, email, role, token, expires_at

projects
  id, workspace_id, title, subject, grade, learner_context (jsonb)

designs
  id, project_id
  phase_T (jsonb)     ← 팀 준비 전체
  phase_A (jsonb)     ← 분석 전체
  phase_Ds (jsonb)    ← 설계 전체
  phase_DI (jsonb)    ← 개발/실행 전체
  phase_E (jsonb)     ← 평가/성찰 전체
  updated_at

versions
  id, design_id, snapshot (jsonb), created_by, created_at

messages
  id, project_id, user_id, role (user / assistant), content, created_at

comments
  id, design_id, section_key, user_id, body, resolved, created_at

activity_logs
  id, project_id, user_id, action_type, section_key
  before (jsonb), after (jsonb), created_at
```

## 5. MVP 범위 및 Phase별 빌드 계획

### Phase 1 — 기반 구축

- 인증 (이메일)
- 워크스페이스 생성 및 팀원 초대
- 프로젝트 생성 및 학습자 맥락 입력
- 팀 준비(T) 단계 설계 캔버스
- AI 채팅 패널 (스트리밍) 기본 연동
- 활동 로그 DB 기록 시작

### Phase 2 — AI 협력 고도화

- 전체 단계(A → Ds → DI → E) 설계 캔버스
- AI 제안 수락·수정·거절 UX
- AI 정합성 자동 점검 (비전↔목표↔평가 체인)
- 버전 스냅샷 (수락 시 자동 저장)

### Phase 3 — 팀 협업 및 출력

- 섹션별 코멘트 스레드 + 알림
- 역할별 UI 권한 분기 (Host / Editor / Viewer)
- Markdown 출력
- PDF 출력 (종합 보고서)

## 6. 주요 설계 결정 사항

**AI 없이도 가능한 구조 유지**
AI 에이전트가 없더라도 교사 팀이 단계별 가이드를 따라 설계를 완료할 수 있도록 모든 섹션에 활동 안내 텍스트를 포함한다. AI는 선택적 보조 수단이다.

**단계 진행 강제 없음**
교사가 임의의 단계부터 시작하거나 단계를 건너뛸 수 있다. 에이전트는 현재 완료된 단계를 기반으로 다음 단계를 안내하되 강제하지 않는다.

**비동기 협력 우선**
MVP에서는 실시간 동시편집보다 비동기 코멘트 + 명시적 저장 방식을 채택한다. 충돌 해결 복잡도를 줄이고 안정성을 확보한다.

**AI 기록의 신뢰성**
AI가 자동으로 저장하는 내용은 반드시 교사의 확인(수락) 후 최종 반영된다. AI 단독으로 설계 내용을 확정하지 않는다.
